# 表单字段灵活配置系统规划

## 一、现状分析

### 1.1 现有系统架构
- **流程引擎**: 已配置化设计，支持不同工单类型、不同业务场景的灵活流程配置
- **表单组件**: DynamicForm.vue 支持多种字段类型（text, textarea, number, select, date, datetime, rate, location, imageUpload, signature, partsSelector, inspectionItems）
- **字段配置**: workflowConfigStore.js 中硬编码了 formFieldConfigs，包含约 30 个预定义字段
- **配置界面**: WorkflowConfig.vue 提供流程模板和节点配置，但字段选择是固定的下拉列表

### 1.2 当前痛点
1. **字段硬编码**: 新增字段需要修改代码，无法动态配置
2. **缺乏字段管理界面**: 没有可视化的字段配置管理功能
3. **字段复用性差**: 同名字段在不同场景下可能需要不同配置
4. **客户定制化困难**: 不同客户需求不同，无法快速响应
5. **版本管理缺失**: 字段变更没有版本控制

## 二、目标与需求

### 2.1 核心目标
实现一个**可视化、可配置、可扩展**的表单字段管理系统，支持：
- 动态字段定义与配置
- 字段分组与分类管理
- 字段级权限控制
- 多版本字段配置
- 客户级字段定制

### 2.2 功能需求

#### 2.2.1 字段类型扩展
| 基础类型 | 业务类型 | 高级类型 |
|---------|---------|---------|
| text | customerSelector | table |
| textarea | engineerSelector | subForm |
| number | assetSelector | computed |
| select | partsSelector | conditional |
| date | location | reference |
| datetime | signature | richText |
| rate | imageUpload | fileUpload |
| checkbox | barcodeScan | multiLang |
| radio | qrCode | json |

#### 2.2.2 字段配置能力
- **基础属性**: 名称、标签、类型、必填、默认值、占位符
- **验证规则**: 必填、长度、格式、范围、正则、自定义函数
- **显示控制**: 可见性条件、禁用条件、只读条件
- **联动规则**: 值变化联动、字段显示联动、选项联动
- **数据源**: 静态选项、API接口、字典数据、关联字段
- **样式配置**: 宽度、布局、CSS类、占位格数

#### 2.2.3 场景适配
- 不同工单类型（维修/安装/巡检/配件销售）使用不同字段集
- 不同客户可定制专属字段
- 不同流程节点显示不同字段
- 不同角色查看/编辑不同字段

### 5.1 字段类型注册机制

```javascript
// fieldTypeRegistry.js
class FieldTypeRegistry {
  constructor() {
    this.types = new Map();
    this.renderers = new Map();
  }
  
  // 注册字段类型
  register(typeDef) {
    this.types.set(typeDef.code, typeDef);
  }
  
  // 注册渲染器
  registerRenderer(typeCode, renderer) {
    this.renderers.set(typeCode, renderer);
  }
  
  // 获取字段类型定义
  getType(code) {
    return this.types.get(code);
  }
  
  // 获取渲染器
  getRenderer(code) {
    return this.renderers.get(code);
  }
  
  // 获取所有类型
  getAllTypes() {
    return Array.from(this.types.values());
  }
  
  // 按分类获取类型
  getTypesByCategory(category) {
    return this.getAllTypes().filter(t => t.category === category);
  }
}

export const fieldTypeRegistry = new FieldTypeRegistry();
```

### 5.3 联动规则引擎

```javascript
// linkageEngine.js
class LinkageEngine {
  constructor() {
    this.rules = [];
    this.context = {};
  }
  
  // 添加规则
  addRule(rule) {
    this.rules.push(rule);
  }
  
  // 设置上下文数据
  setContext(context) {
    this.context = context;
  }
  
  // 评估条件
  evaluateCondition(condition) {
    // 支持的条件表达式：
    // - {{fieldName}} == 'value'
    // - {{fieldName}} != 'value'
    // - {{fieldName}} > 10
    // - {{fieldName}} in ['a', 'b']
    // - {{field1}} && {{field2}}
    
    const expr = condition.replace(/\{\{(\w+)\}\}/g, (match, fieldName) => {
      const value = this.context[fieldName];
      return JSON.stringify(value);
    });
    
    try {
      return new Function('return ' + expr)();
    } catch (e) {
      console.warn('Condition evaluation error:', e);
      return false;
    }
  }
  
  // 执行联动
  execute(fieldName, value) {
    this.context[fieldName] = value;
    
    this.rules.forEach(rule => {
      if (rule.trigger === fieldName || rule.trigger === '*') {
        const shouldExecute = this.evaluateCondition(rule.condition);
        if (shouldExecute) {
          this.applyAction(rule.action);
        }
      }
    });
  }
  
  // 应用动作
  applyAction(action) {
    switch (action.type) {
      case 'setValue':
        this.context[action.target] = action.value;
        break;
      case 'show':
        // 触发显示事件
        break;
      case 'hide':
        // 触发隐藏事件
        break;
      case 'enable':
        // 触发启用事件
        break;
      case 'disable':
        // 触发禁用事件
        break;
      case 'setOptions':
        // 更新选项
        break;
    }
  }
}
```

### 5.4 配置继承与合并

```javascript
// configMerger.js
class ConfigMerger {
  // 合并字段定义与实例覆盖配置
  mergeFieldConfig(fieldDef, fieldInstance) {
    return {
      ...fieldDef,
      config: {
        ...fieldDef.config,
        ...fieldInstance.override
      },
      validation: fieldInstance.override?.validation || fieldDef.validation,
      displayControl: fieldInstance.override?.displayControl || fieldDef.displayControl,
      style: {
        ...fieldDef.style,
        ...fieldInstance.config
      }
    };
  }
  
  // 合并客户定制配置
  mergeCustomerConfig(baseTemplate, customerConfig) {
    const mergedFields = [...baseTemplate.fields];
    
    // 应用移除
    const removedIds = new Set(customerConfig.customizations.removeFields);
    const filteredFields = mergedFields.filter(f => !removedIds.has(f.id));
    
    // 应用修改
    customerConfig.customizations.modifyFields.forEach(override => {
      const field = filteredFields.find(f => f.id === override.fieldId);
      if (field) {
        Object.assign(field.override, override.changes);
      }
    });
    
    // 应用新增
    filteredFields.push(...customerConfig.customizations.addFields);
    
    // 重新排序
    filteredFields.sort((a, b) => (a.config?.sortOrder || 0) - (b.config?.sortOrder || 0));
    
    return {
      ...baseTemplate,
      fields: filteredFields
    };
  }
}
```

## 十一、开发日志

### 2026-04-03 - Bug修复

#### 问题描述
运行时出现 `ComputedRefImpl` 错误：
```
fieldConfigStore.allFields is not an array: ComputedRefImpl {setter: undefined, _value: undefined, dep: Dep, __v_isRef: true, fn: ƒ, …}
```

#### 问题原因
Store 中直接导出 ref 和 computed 对象，在组件中访问时 Vue 3 的响应式系统在某些情况下不会自动解包，导致获取到的是 `ComputedRefImpl` 对象而非实际值。

#### 修复方案
将 Store 中的 State 和 Getters 改为使用 getter 函数返回实际值：

**修改文件**:
1. `src/stores/fieldConfigStore.js`
   - `fieldDefinitions` → `get fieldDefinitions() { return fieldDefinitions.value }`
   - `fieldGroups` → `get fieldGroups() { return fieldGroups.value }`
   - `allFields` → `get allFields() { return allFields.value }`
   - `activeFields` → `get activeFields() { return activeFields.value }`

2. `src/stores/fieldTemplateStore.js`
   - `templates` → `get templates() { return templates.value }`
   - `customerConfigs` → `get customerConfigs() { return customerConfigs.value }`
   - `allTemplates` → `get allTemplates() { return allTemplates.value }`
   - `activeTemplates` → `get activeTemplates() { return activeTemplates.value }`
   - `defaultTemplates` → `get defaultTemplates() { return defaultTemplates.value }`

3. `src/stores/formLayoutStore.js`
   - `layouts` → `get layouts() { return layouts.value }`
   - `allLayouts` → `get allLayouts() { return allLayouts.value }`

#### 修复结果
- 组件中可以直接访问 `fieldConfigStore.fieldDefinitions` 获取数组数据
- 无需在组件中使用 `.value` 解包
- 避免了 `ComputedRefImpl` 相关的运行时错误

### 2026-04-04 - 字段管理页面白屏修复

#### 问题描述
字段管理页面出现白屏，无法正常显示字段列表。

#### 问题原因
1. **Store 中的 ref 未正确导出**: `loading` 和 `currentField` 等 ref 直接导出，导致在模板中使用时获取到的是 ref 对象而非实际值
2. **动态图标组件未导入**: 使用 `<component :is="getFieldTypeIcon(row.type)" />` 动态渲染图标，但返回的图标名称字符串没有被正确映射到实际的图标组件
3. **可选链访问问题**: `row.config.required` 访问时 `row.config` 可能为 undefined

#### 修复方案

**1. 修复 Store 导出** (`src/stores/fieldConfigStore.js`, `src/stores/fieldTemplateStore.js`, `src/stores/formLayoutStore.js`):
- `loading` → `get loading() { return loading.value }`
- `currentField` → `get currentField() { return currentField.value }`
- `currentTemplate` → `get currentTemplate() { return currentTemplate.value }`
- `currentLayout` → `get currentLayout() { return currentLayout.value }`

**2. 修复图标导入** (`src/views/FieldConfigManager.vue`):
- 导入所有需要的 Element Plus 图标组件
- 创建图标映射表 `iconMap` 将字符串名称映射到组件
- 修改 `getFieldTypeIcon` 方法返回实际的组件而非字符串

**3. 修复可选链访问** (`src/views/FieldConfigManager.vue`):
- `v-model="row.config.required"` → `:model-value="row.config?.required"`

#### 修复结果
- 字段管理页面可以正常显示字段列表
- 图标正确渲染
- 页面不再出现白屏
